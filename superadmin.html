Introduction

The CRM Pro super admin panel is a single-page React application that gives the super admin complete control over users, roles, permissions, teams, and manager assignments. The super admin is the highest-privilege account — their RBAC role carries all 45 seeded permissions.

ℹ️
There is no separate Super Admin database model. The super admin is a regular User whose assigned RBAC role (typically named superadmin) carries every permission. This role is created automatically by the database seed script.

What the Super Admin can do

DomainActions
AuthLogin, view own profile, refresh token, logout
Users / EmployeesCreate users, list all users, view by ID, assign roles at creation
RolesCreate, read, update, delete RBAC roles
PermissionsCreate, read, update, delete permissions; assign/remove from roles
TeamsCreate, edit, delete teams; assign/remove members; set team lead
ManagersAssign one or more teams to any manager user

Axios Instance (src/api/axios.js)

All API calls go through a single shared Axios instance configured at startup:

const api = axios.create({
  baseURL: import.meta.env.VITE_BASE_APP_URL,  // e.g. http://localhost:4000/api
  headers: { 'Content-Type': 'application/json' },
})

// Request interceptor — attaches Bearer token from localStorage
api.interceptors.request.use(config => {
  const token = localStorage.getItem('accessToken')
  if (token) config.headers.Authorization = `Bearer ${token}`
  return config
})

// Response interceptor — on 401, clears auth and redirects to /login
api.interceptors.response.use(
  response => response,
  error => {
    if (error.response?.status === 401) {
      localStorage.removeItem('accessToken')
      localStorage.removeItem('user')
      window.location.href = '/login'
    }
    return Promise.reject(error)
  }
)

Auth Flow

Login

  1. User opens /login. A gradient split-screen form is shown (logo left, form right on desktop; stacked on mobile).
  2. User enters Email (required, valid email) and Password (required, min 6 chars). A show/hide toggle is on the password field.
  3. On submit → POST /api/auth/login with { email, password }.
  4. On success: accessToken and user object are saved to localStorage. User is redirected to /dashboard.
  5. On error: an inline red error banner is shown below the form header.

Validation (client-side)

FieldRule
emailRequired, valid email format
passwordRequired

Session Persistence

AuthContext initialises user and isAuthenticated from localStorage on every page load, so the session survives hard refreshes without calling the backend.

// AuthContext init
const storedUser = localStorage.getItem('user')
const storedToken = localStorage.getItem('accessToken')
// Both must exist for isAuthenticated = true

Logout

  1. User clicks the logout icon in the bottom of the Sidebar.
  2. logout() from AuthContext clears accessToken and user from localStorage.
  3. User is programmatically redirected to /login via React Router's useNavigate.
ℹ️
The frontend does not currently call POST /api/auth/logout — it only clears local storage. Add this call if you need server-side refresh token invalidation.

Managers Page

Overview

Route: /managers. Displays all managers with their currently assigned teams. Super admin can assign or unassign teams from each manager.

Manager Detection Logic

A "manager" is any user whose role name contains "manager" (case-insensitive substring match). If no such users are found, the page falls back to showing all users, with an amber warning banner.

⚠️
Banner when no manager role exists: "No users with a Manager role found. Showing all users — assign a role containing "manager" to an employee to designate them as a manager."

On Mount

Fetches in parallel:

  • GET /api/users → to identify managers
  • GET /api/teams → to know which teams exist and which are assigned

After fetch, teams are passed through applyStoredAssignments() which merges localStorage data (see Persistence section below).

  • Coloured top accent bar (6-colour deterministic from first char of manager name)
  • Avatar square with initials + name + email + role badge pills (violet)

Team Assignment Persistence

Since GET /api/teams does not return a managerId field, assignments are stored in localStorage under the key crm_manager_team_assignments as a flat map:

// localStorage value
{
  "<teamId>": "<managerId>",
  "<teamId2>": "<managerId>"
}

On every page load, this map is overlaid onto the fresh API response before setting state. On every save, the map is updated atomically with the new assignments.

ℹ️
If your backend adds a managerId field to the Team model and returns it in GET /api/teams, remove the localStorage layer — the field will be picked up directly.

Assign Teams Modal

Trigger

Clicking Manage Teams on any manager card.

Layout

Violet gradient header, manager info chip (name, email, live "N selected" counter), search bar, scrollable team checklist, footer with Cancel + Save Changes.

Team List

Every team is shown as a button with:

  • Team name + member count
  • Square checkbox (violet when selected)
  • If the team is already assigned to a different manager: shown as dimmed (opacity-50) with "Other mgr" amber label — cannot be checked

Pre-selection

Teams already assigned to this manager are pre-checked using currentTeamIds (derived from getManagerTeams(managerId) which filters teams by team.managerId === managerId).

Save Flow

  1. Computes toAdd = newly checked IDs not in original set.
  2. Computes toRemove = originally checked IDs now unchecked.
  3. Fires all changes in parallel: PATCH /api/teams/:id with { managerId } for additions, { managerId: null } for removals.
  4. On success: onSaved(assignedIds) → parent updates local state + localStorage + shows success toast.

Save button disabled when

No changes detected (selected set equals original set), or a save is in progress.
